# 构建高效上下文感知多智能体框架

**作者：Hangfei Lin，技术负责人 | 2025年12月4日**

AI 智能体开发领域正在快速变革。各类组织正在部署自主智能体来处理长时序任务：工作流自动化、深度研究、维护复杂代码库等。这种雄心立即遭遇了一个瓶颈：上下文。

随着智能体运行时间延长，它们需要追踪的信息量——聊天历史、工具输出、外部文档、中间推理过程——呈爆炸式增长。业界盛行的"解决方案"是依赖越来越大的上下文窗口。但仅仅给智能体更多文本空间不可能成为唯一的扩展策略。

## 扩展瓶颈

naive 模式——将所有内容追加到一个巨大的 prompt 中——在三方面压力下崩溃：

1. **成本与延迟螺旋上升**：模型成本和首 token 响应时间随上下文大小快速增长。

2. **信号退化（"中间迷失"）**：充满无关日志的上下文窗口会干扰模型，导致它执着于过去的模式而非眼前的指令。

3. **物理限制**：实际工作负载最终会溢出即使最大的固定窗口。

## 设计理念：上下文即编译视图

ADK 建立在不同的理念之上：**上下文是一个更丰富的有状态系统的编译视图**。

在这个视图下：
- **会话、记忆和产物**是来源——交互及其数据的完整结构化状态。
- **流程和处理器**是编译管道——一系列对状态进行转换的传递过程。
- **工作上下文**是编译视图，为本次调用发送给 LLM。

三条设计原则指导这一架构：
- **存储与呈现分离**：区分持久化状态（会话）和单次调用视图（工作上下文）。
- **显式转换**：上下文通过命名有序的处理器构建，而非临时字符串拼接。
- **按需作用域**：每个模型调用和子智能体只看到所需的最小上下文。

## 1. 结构：分层模型

ADK 将上下文组织为不同的层级：

- **工作上下文** – 本次模型调用的即时 prompt
- **会话** – 作为结构化 Event 对象记录的交互持久日志
- **记忆** – 长寿命、可搜索的知识，超越单个会话存在
- **产物** – 通过名称和版本寻址的大型二进制或文本数据

### 1.1 工作上下文作为重新计算的视图

每次调用时，ADK 从底层状态重建工作上下文。此视图是临时的、可配置的，且与模型无关。

### 1.2 流程和处理器：上下文处理即管道

每个基于 LLM 的智能体都背靠一个 LLM Flow，维护有序的处理器列表：

```python
self.request_processors += [
    basic.request_processor,
    auth_preprocessor.request_processor,
    request_confirmation.request_processor,
    instructions.request_processor,
    identity.request_processor,
    contents.request_processor,
    context_cache_processor.request_processor,
    planning.request_processor,
    code_execution.request_processor,
    output_schema_processor.request_processor,
]

self.response_processors += [
    planning.response_processor,
    code_execution.response_processor,
]
```

### 1.3 会话和事件：结构化、语言无关的历史

ADK 将每次交互捕获为强类型 Event 记录。这提供了：
- **模型无关性**：在不重写历史的情况下切换底层模型
- **丰富操作**：下游组件在丰富的事件流上操作
- **可观测性**：天然的分析和调试界面

`contents` 处理器执行选择、转换和注入，桥接会话和工作上下文。

### 1.4 上下文压缩与过滤

ADK 的上下文压缩功能在达到阈值时触发异步摘要处理。它使用 LLM 在滑动窗口上摘要旧事件，并将结果作为新事件写回。

收益逐级向下传递：
- **可扩展性**：会话在长时间对话中保持可管理
- **清晰的视图**：contents 处理器在已压缩的历史上工作
- **解耦**：在不触及智能体代码的情况下调整压缩策略

### 1.5 上下文缓存

现代模型支持上下文缓存，允许推理引擎复用注意力计算。ADK 将上下文窗口划分为：
- **稳定前缀**：系统指令、智能体身份、长寿命摘要
- **可变后缀**：最新用户轮次、新工具输出

`static instruction` 原语保证系统 prompt 的不可变性，确保缓存前缀有效。

## 2. 相关性：智能体管理当下重要的事

最优工作上下文是人类定义架构与智能体决策之间的协商。

### 2.1 产物：外部化大型状态

ADK 将大型数据视为**产物**：由 `ArtifactService` 管理的命名版本化对象。句柄模式将大型数据保留在产物存储中，而非 prompt 中。智能体仅在需要原始数据时使用 `LoadArtifactsTool`。

ADK 支持**临时扩展**：一旦模型调用或任务完成，产物默认从工作上下文中卸载。

### 2.2 记忆：按需检索的长期知识

`MemoryService` 通过两种模式管理长寿命、语义化的知识：
- **反应式召回**：智能体在识别到知识缺口时显式调用 `load_memory_tool`
- **主动召回**：系统预处理器在调用前通过 `preload_memory_tool` 注入相关片段

## 3. 多智能体上下文：谁在何时看到什么

多智能体系统放大了上下文膨胀。ADK 允许你显式限定被调用者能看到什么，抑制大部分祖先历史。

### 3.1 两种多智能体交互模式

1. **智能体即工具**：根智能体将专业智能体视为函数——用聚焦的 prompt 调用，获取结果，然后继续。

2. **智能体转移（层级）**：控制权完全移交给子智能体继续对话。

### 3.2 智能体转移的受限切换

交接行为由 `include_contents` 等旋钮控制。默认模式传递完整内容；`none` 模式不传递历史。专业智能体接收所需的最少上下文。

### 3.3 智能体转移时的对话翻译

转移控制时，ADK 执行主动翻译：
- **叙事转换**：之前的"Assistant"消息重新转换为叙事上下文
- **行动归属**：工具调用被标记或摘要，使新智能体能够基于结果行动，而不会将执行与自身能力混淆

## 结论

上下文管理不能再意味着字符串操作。它必须被视为与存储和计算并列的架构关注点。

ADK 的上下文架构——分层存储、编译视图、管道处理和严格作用域——封装了将智能体从原型推进到可扩展、可靠的生产系统所需的严格系统工程。
